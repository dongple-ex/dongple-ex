# 내발문자 설계 - 핵심 사용자 흐름

## 문제 정의
현재 FD는 `홈`, `소식`, `기록`, `지도`, `나의 내발문자`를 각각 설명하지만, 사용자가 앱 안에서 왜 다음 화면으로 이동해야 하는지에 대한 하나의 생활 루프가 약하다.

사용자가 기대하는 자연스러운 흐름은 다음과 같다.

1. 홈에서 지금 지역의 방문 계기를 발견한다.
2. 소식에서 행사/장소의 맥락을 읽는다.
3. 기록으로 지금 본 것, 다녀온 것, 확인한 것을 남긴다.
4. 지도에서 그 기록이 위치와 현재 상태로 연결된다.
5. 나의 내발문자에서 기억하고 싶은 장소와 기록을 다시 관리한다.

이 흐름이 끊기면 내발문자는 `지도 앱`, `소식 앱`, `기록장`, `앨범`이 따로 있는 것처럼 느껴진다.

## 핵심 루프

### 발견
- 진입 화면: `홈`
- 사용자가 얻는 것:
  - 지금 핫한 곳
  - 오늘 행사
  - 내 주변 실시간 상태
  - 최근 소식의 단서
- 다음 행동:
  - `지도에서 보기`
  - `소식 자세히 보기`
  - `기록 남기기`

### 맥락 이해
- 진입 화면: `소식`
- 사용자가 얻는 것:
  - 공식 행사/장소 요약
  - 방문 전후의 지역 소식
  - 방문 전 참고할 정보
- 다음 행동:
  - `지도에서 위치 보기`
  - `기록으로 남기기`
  - `나의 내발문자에 기억하기`

### 현재화
- 진입 화면: `지도`
- 사용자가 얻는 것:
  - 위치 기반 현재 상태
  - 행사/장소의 붐빔, 대기, 분위기
  - 내가 남긴 기록의 위치 맥락
- 다음 행동:
  - `현장 공유`
  - `상황이 달라요`
  - `상세 보기`
  - `나의 내발문자에 저장`

### 기록
- 진입 화면: `기록`
- 사용자가 남기는 것:
  - 상황공유: 지금 상태, 위치, 분위기
  - 소식 작성: 경험, 질문, 추천, 후기
- 저장 결과:
  - 상황공유는 지도 상태와 나의 내발문자에 반영된다.
  - 소식 작성은 소식 피드와 나의 내발문자에 반영된다.

### 기억
- 진입 화면: `나의 내발문자`
- 사용자가 얻는 것:
  - 내가 남긴 기록
  - 기억한 장소
  - 즐겨찾기
  - 다시 방문할 후보
- 다음 행동:
  - `지도에서 다시 보기`
  - `소식으로 다시 보기`
  - `새 기록 남기기`

## 화면별 역할 재정의

### 홈
홈은 단순 대시보드가 아니라 `오늘 무엇을 볼지`를 정하는 시작점이다.

홈의 우선순위:
1. 지금 상태 요약
2. 오늘 행사/핫플 단서
3. 지도 진입
4. 기록 진입
5. 소식 미리보기
6. 나의 내발문자 회고 단서

홈에서 소식은 피드 자체가 아니라 `더 볼 이유`가 되어야 한다.

### 소식
소식은 커뮤니티 피드이지만, 공식 행사/장소 정보가 섞일 때는 `읽고 끝`이 아니라 다음 행동을 제안해야 한다.

소식 상세의 필수 후속 행동:
- `지도에서 위치 보기`
- `기록으로 남기기`
- `나의 내발문자에 기억하기`

공식 정보 상세는 API 필드 나열이 아니라 `지금 볼 요약`, `가기 전에 확인`, `현장 안내`처럼 방문 판단을 돕는 요약이어야 한다.

### 기록
기록은 별도 글쓰기 화면이 아니라 사용자의 경험을 내발문자 현장 데이터로 바꾸는 변환 지점이다.

기록 완료 후 후속 흐름:
- 상황공유 완료: `지도에서 반영 보기`, `나의 내발문자에 저장됨`
- 소식 작성 완료: `소식에서 보기`, `나의 내발문자에 저장됨`

작성 후 그냥 닫히면 기록의 가치가 약해진다.

### 지도
지도는 위치 확인 화면이 아니라 기록이 살아나는 화면이다.

지도에서 보여야 하는 것:
- 지금 상태
- 내가 남긴/기억한 장소
- 공식 행사와 사용자 현장 상태의 연결
- 소식에서 본 장소의 위치

지도 액션:
- `현장 공유`
- `상황이 달라요`
- `기억하기`
- `나의 내발문자에서 보기`

### 나의 내발문자
나의 내발문자는 단순 앨범이 아니라 `내 기록이 다음 방문으로 이어지는 곳`이다.

필수 CTA:
- `지도에서 다시 보기`
- `이 장소 소식 보기`
- `새 기록 남기기`
- `즐겨찾기`

## 정보 구조 원칙

### 기록의 단위
모든 사용자 기록은 가능한 한 장소와 연결한다.

기록 저장 시 권장 필드:
- `type`: status | post | favorite
- `source_id`
- `title`
- `description`
- `place_name`
- `address`
- `latitude`
- `longitude`
- `category`
- `status_label`
- `created_at`
- `tourapi_content_id` 또는 공식 장소 매칭 ID

위치가 없는 기록은 내발문자에서 회고는 가능하지만 지도 재진입이 약해진다.

### 공식 데이터의 역할
공식 데이터는 사용자의 기록을 대체하지 않는다.

공식 데이터는 다음 역할로 제한한다.
- 소식 상세의 요약 보강
- 지도 위치 뼈대
- 내발문자의 재방문 정보 보강

내발문자의 핵심 가치는 공식 데이터 자체가 아니라, 공식 장소 위에 얹히는 사용자의 현재 상태와 개인 기록이다.

## 권장 사용자 여정

### 여정 A: 소식에서 발견하고 지도에서 확인
1. 홈에서 오늘 행사/추천 장소를 본다.
2. 소식 상세에서 `지금 볼 요약`을 읽는다.
3. `지도에서 위치 보기`를 누른다.
4. 지도에서 현재 상태와 주변 순간을 확인한다.
5. 현장에 있으면 `현장 공유`를 남긴다.
6. 기록은 나의 내발문자에 쌓인다.

### 여정 B: 지도에서 발견하고 기록으로 남김
1. 홈에서 `지도 바로 보기`로 이동한다.
2. 지도에서 핫한 장소나 행사 핀을 선택한다.
3. 현재 상태를 본다.
4. `여기는요!` 또는 `상황이 달라요`로 기록한다.
5. 기록은 지도 카드와 나의 내발문자에 반영된다.

### 여정 C: 내발문자에서 다시 방문
1. 나의 내발문자에서 즐겨찾기 또는 발자취를 본다.
2. 기억한 장소를 선택한다.
3. `지도에서 다시 보기`로 현재 상태를 확인한다.
4. 필요하면 새 기록을 남긴다.

## 현재 구현과의 차이

### 부족한 점
- 홈이 소식/기록/지도/나의 내발문자를 하나의 루프로 안내하지 못한다.
- 소식 상세에 `지도에서 위치 보기`, `기록으로 남기기`, `기억하기`가 약하다.
- 기록 완료 후 `지도에 반영됨`, `나의 내발문자에 저장됨` 피드백이 약하다.
- 지도에서 내발문자 저장/기억하기 흐름이 약하다.
- 나의 내발문자 기록이 지도 재진입과 강하게 연결되지 않는다.
- `앨범` 구현은 있지만 FD 기준 명칭인 `내발문자`의 상위 개념과 아직 완전히 맞지 않는다.

### 우선 보완 순서
1. 소식 상세에 `지도에서 위치 보기`, `기록으로 남기기`, `기억하기` CTA를 추가한다.
2. 기록 완료 후 `지도에서 보기`, `나의 내발문자에서 보기` 후속 행동을 보여준다.
3. 나의 내발문자 메모리에 좌표와 주소를 포함한다.
4. 내발문자 카드에 `지도에서 다시 보기`를 붙인다.
5. 홈에 `오늘 볼 것`, `지금 확인할 것`, `나중에 기억할 것`의 세 단서를 연결한다.

## 성공 기준
- 사용자가 홈에서 소식을 보고 끝나지 않고 지도 또는 기록으로 자연스럽게 이동한다.
- 사용자가 기록을 남기면 그 기록이 지도와 나의 내발문자에 반영된다는 것을 즉시 이해한다.
- 나의 내발문자에서 기억한 장소가 다시 지도 확인으로 이어진다.
- 내발문자가 단발성 정보 앱이 아니라 `발견하고, 확인하고, 남기고, 다시 꺼내보는 앱`으로 느껴진다.
