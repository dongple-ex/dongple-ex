## 공통 원칙
- 홈은 `상황 중심 허브`다.
- 지도는 `실시간 탐색과 공유의 핵심 화면`이다.
- 기록은 `상황공유`와 `소식 작성`을 아우르는 공통 진입점이다.
- 소식은 `커뮤니티형 게시판`이다.
- 나의 동플은 `기록의 축적, 회고, 재방문`을 다루는 개인 영역이다.

## 구현 아키텍처 원칙
- `app/*`는 조립과 라우팅의 책임을 가진다.
- 실제 도메인 로직은 `features/*`와 `services/*`로 분리한다.
- UI 컴포넌트와 데이터 접근 로직은 분리한다.
- 공식 데이터와 사용자 데이터는 같은 화면에 보여도 저장 축은 분리한다.

## 전역 내비게이션 원칙
- 하단 메뉴의 기준 축은 `홈 / 지도 / 기록 / 소식 / 나의 동플`이다.
- `기록`은 단일 기능이 아니라 `상황공유`와 `소식 작성`으로 이어지는 허브다.
- `나의 동플`은 개인 기능의 상위 개념이며, 앨범은 그 안의 주요 섹션이다.

## 브랜드 확장 원칙
- `동플`은 코어 서비스 브랜드다.
- 지역 론칭 시에는 `동플: 수원에 가다`, `동플: 강원삼척에 가다`처럼 지역 테마를 얹을 수 있다.
- 지역 테마는 브랜딩과 큐레이션 레이어이며, 제품 구조 자체는 공통 코어를 유지한다.

## 공공데이터 원칙
- 한국관광공사 OpenAPI는 공식 관광 데이터 축이다.
- 사용자 생성 데이터와 공공데이터는 같은 화면에 공존할 수 있지만 성격은 분리해서 설명한다.
- 운영 기준은 [동플_TourAPI_연동운영_가이드.md](/C:/my_project/dongple_ex/FD/동플_TourAPI_연동운영_가이드.md)를 따른다.

## 문서 운영 규칙
- 새로운 기능은 먼저 어느 메뉴 축에 속하는지 결정한 뒤 해당 설계 문서에 반영한다.
- 화면 수정이 메뉴 경계를 넘으면 인덱스와 관련 설계 문서를 함께 갱신한다.
- 이전 구상 문서는 삭제하지 않고 참고용으로 유지한다.

## 기존 문서에서 흡수한 핵심 항목
- 프론트 구조 문서의 `app / features / services` 분리 원칙
- 시스템 설계 문서의 `상황공유 / 커뮤니티 / 공식 이벤트 / 인증 / 운영` 모듈 구분
- 지도/상태보드 보완 문서의 `바텀시트`, `드래그`, `지도-리스트 연동`, `즉시 액션` UX
- TourAPI 문서의 `Ground Truth`, `fallback`, `메타 보강`, `나의 동플` 확장 관점
