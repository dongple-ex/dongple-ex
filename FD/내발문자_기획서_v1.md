# 내발문자 기획서 v1

## 한 줄 정의
**내 발이 남긴 문장, 다음 발걸음의 선택이 된다.**

## 서비스 개요
내발문자는 현장의 `지금`을 공유해 사람들의 선택을 바꾸는 실시간 상황 네트워크다.

내발문자는 `dongple_ex`에서 출발한 동플의 확장 실험을 별도 브랜드로 재정의한 이름이다. 동플이 지역 실시간 상황 공유의 가능성을 확인하는 초기 제품이었다면, 내발문자는 사용자의 발걸음, 현장 기록, 관광 공공데이터를 연결해 `방문 전 판단`과 `방문 후 기억`을 함께 다루는 확장 버전이다.

사용자는 장소에 가기 전 늘 같은 질문을 한다.

- 지금 사람 많을까?
- 대기 길까?
- 분위기 어떨까?

기존 서비스는 과거 리뷰와 평균 데이터 중심이라 `지금 이 순간`을 알려주지 못한다. 내발문자는 이 공백을 해결한다.

## 핵심 가치
- **실시간성**: 과거가 아니라 지금을 보여준다.
- **간결함**: 3초 안에 보고, 3초 안에 공유한다.
- **행동 변화**: 정보를 주는 데서 끝나지 않고 사용자의 결정을 바꾼다.

## 핵심 사용자 경험
앱을 켜는 순간 사용자는 이렇게 느껴야 한다.

> 오, 지금 여기 상황 바로 보이네?

그리고 곧바로 다음 판단으로 이어져야 한다.

> 이거 보고 결정해야겠다.

## MVP 주요 기능

- 주변 장소 혼잡도 표시

### 2. 상황 공유

### 4. 나의 내발문자

- 영향력 점수

## UX 구조
첫 화면은 무조건 `내 주변 실시간 상태`다.

- 우선: 지도 + 상태
- 제외: 찜 목록 중심 첫 화면
- 제외: 커뮤니티 글 중심 첫 화면

공유 UX는 최대 3초를 기준으로 한다.

- 버튼 중심
- 텍스트 최소화
- 공유 직후 지도/상태에 즉시 반영

## 차별화 전략

| 요소 | 기존 서비스 | 내발문자 |
| --- | --- | --- |
| 시간 | 과거/평균 | 지금 |
| 데이터 | 리뷰/로그 | 실시간 입력 |
| 목적 | 정보 | 행동 |
| 속도 | 느림 | 즉시 |

내발문자의 핵심 포지션은 `타이밍을 알려주는 앱`, `확인하는 앱`, `지금 어디가 살아있는지 보여주는 앱`이다.

## 행사와 일상 전략
내발문자는 행사만 다루는 앱이 아니다. 행사만 있으면 평소에 쓸 이유가 없어지고, 행사가 없는 날에는 앱이 죽는다. 반대로 일반 장소만 다루면 초기 트래픽을 만들기 어렵다.

따라서 구조는 다음처럼 잡는다.

- **행사 = 진입점**
- **일상 = 확장**

행사는 초기에 사람이 몰리고 이야깃거리가 생기는 강한 진입점이다. 일상 장소는 사용자가 평소에도 계속 확인하게 만드는 기본 사용 이유다.

### 1. 행사 레이어
- 행사 핀을 강하게 강조한다.
- 행사 일정 자체보다 `지금 붐빔`, `대기 길다`, `입장 원활`, `주변 혼잡` 같은 현재 상태를 보여준다.
- 사람 몰리는 곳을 중심으로 초기 핵심 트래픽을 만든다.
- 콘텐츠화가 쉬워 바이럴에 유리하다.
  - 예: `오늘 수원화성 주변 상황`
  - 예: `지금 수원연극축제 현장 붐빔`

### 2. 일반 장소 레이어
- 관광지 주변 카페, 음식점, 공원, 편의시설 같은 방문 맥락 장소를 기본 레이어로 둔다.
- 행사가 없는 날에도 지역 관광 동선과 생활권 상태를 확인할 이유를 만든다.
- 반복 조회와 지역 현장 공유의 기반이 된다.

### UX 전략
첫 화면은 `지금 핫한 곳`을 먼저 보여준다.

- 오늘 행사
- 지금 가장 붐비는 곳
- 근처 한산한 곳
- 관광지 주변 카페/식당/공원 같은 기본 장소

필터는 최소한 다음 축을 제공한다.

- 전체
- 행사
- 카페
- 식당
- 공원

### 반드시 지킬 방향
- 행사 정보 자체를 주인공으로 만들지 않는다.
- 행사의 `현재 상태`를 주인공으로 만든다.
- 내발문자는 행사 일정 앱이나 행사 정보 앱이 아니다.
- 내발문자는 `지금 행사 어떤 상태냐`를 알려주는 앱이다.

### 행사 정보 + 실시간 상태 결합
행사 버전의 내발문자는 두 데이터 축을 결합한다.

1. **행사 데이터(공공 API)**
   - 이름
   - 위치
   - 시간
   - 설명
   - 역할: 기본 뼈대

2. **내발문자 현장 데이터**
   - 지금 붐빔
   - 대기 상황
   - 분위기
   - 최근 공유 시각
   - 역할: 진짜 가치

기존 앱은 `벚꽃축제 진행 중`을 알려준다. 내발문자는 `벚꽃축제 - 지금 붐빔(5분 전), 줄 길어요, 사람 많음`처럼 판단 가능한 현재 상태를 보여준다.

핵심 구조는 다음과 같다.

- **공공데이터 = 뼈대**
- **내발문자 = 실시간 뇌**

지도에서는 행사 핀을 강조하고, 색상으로 혼잡도를 보여주며, `몇 분 전` 업데이트인지 함께 표시한다. 사용자는 지도를 보는 순간 갈지, 기다릴지, 피할지 판단해야 한다.

후속 기능으로 `행사 상태 요약`을 제공할 수 있다.

- 지금 매우 붐빔
- 추천 시간: 저녁 8시 이후
- 지금 가장 붐비는 축제

이 구조는 콘텐츠 자동 생성과 성장 전략에도 연결된다.

- 오늘 수원 행사 상황
- 지금 가장 붐비는 축제
- 지금 행궁동 주변 몰림

기술적으로는 행사 API 연동, `events` 저장, 지도 행사 핀 표시, `event_id` 기준 리포트 연결이 필요하다.

**행사를 보여주는 게 아니라, 행사의 `지금 상태`를 보여준다.**

### 최종 구조
**행사 = 불 붙이는 장작, 일상 = 계속 타는 불.**

강력한 한 줄은 다음으로 확장한다.

**지금 가장 핫한 곳을 보여준다.**

## 수익화 구조
- 장소 프로모션: 가게 노출 광고, `지금 한산함` 강조
- 프리미엄 기능: 혼잡 예측, 한산 시간 추천
- 예약 연결: 자리 있음 → 예약, 수수료
- 데이터 판매: 유동 인구 흐름, 상권 분석

## 유저 성장 전략
초기 전략은 한 지역 집중이다.

- 직접 데이터 채우기
- 친구 참여
- 하루 20~30개 등록
- 목표: 앱을 켜면 정보가 있는 상태 만들기

## 보상 구조
보상은 많이 올리는 사람이 아니라 정확하게 올리는 사람에게 집중한다.

- 핵심 개념: 영향력 점수
- 의미: 내 정보로 몇 명이 움직였는가
- 공유 직후 즉시 보상보다 나중에 영향 피드백을 제공한다.

## 기술 구조
- 프론트: Next.js
- 백엔드: Supabase
- 상태관리: TanStack Query
- 지도: 네이버/카카오
- 앱화: Capacitor

전략은 웹으로 먼저 만들고, 이후 앱으로 확장한다.

## 데이터 구조
핵심 데이터 축은 다음 네 축을 기준으로 한다.

- 사용자 축
  - 현재는 Supabase Auth 사용자 또는 익명/닉네임 작성자 정보로 처리한다.
  - 영향력 점수, 배지, 신뢰도, 프로필이 필요해지면 `public.profiles` 테이블을 추가한다.
  - 생성 SQL은 [scripts/create-profiles.sql](/C:/my_project/dongple_ex/scripts/create-profiles.sql)을 기준으로 한다.
- `places`
- `live_status`
- `status_verifications`

핵심 로직은 `live_status`의 최근 30분 데이터를 사용해 평균 혼잡도를 계산하고, 태그 빈도 기반으로 현재 상태를 표시하는 것이다. 장소별 현재 상태 요약이 필요하면 `place_status`는 별도 테이블이 아니라 집계 결과 또는 향후 캐시 테이블로 다룬다.
