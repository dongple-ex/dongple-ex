# 내발문자 설계 - 나의 내발문자

## 역할
- 개인 기록 아카이브
- 내가 남긴 상황공유와 소식을 다시 돌아보고 재방문 가치를 만드는 공간

## 상위 개념
- `나의 내발문자`는 개인 기능의 상위 메뉴다.
- 현재 1차 구현에서는 `앨범/발자취`가 중심이지만, 장기적으로는 아래를 포함한다.
  - 프로필 요약
  - 나의 인증/신뢰도 정보
  - 알림 설정
  - 앨범과 발자취

## 핵심 목표
- 사용자가 좋았던 장소를 다시 기억할 수 있게 한다.
- 기록이 일회성으로 사라지지 않고 축적되도록 한다.
- 발자취와 즐겨찾기를 통해 지속 가능한 관광 기록 경험을 만든다.

## 주요 구성
- 상단 요약
  - 누적 기록 수
  - 기억한 장소 수
  - 즐겨찾기 수
  - 함께한 시간
- 좋았던 곳 다시 보기
  - 대표 메모리 카드
- 즐겨찾기
  - 하트 기반 재방문 후보
- 기록 목록
  - 전체
  - 상황공유
  - 소식
  - 즐겨찾기
- 발자취
  - 시간순 타임라인

## 심리적 목표
- 내 기록이 단순 저장이 아니라 `내가 쌓아가는 지역 관광 경험`처럼 느껴져야 한다.
- 좋았던 장소를 다시 꺼내보는 감정선이 있어야 한다.
- 공유와 재방문이 자연스럽게 이어져야 한다.

## 데이터 구조
- 입력 원천
  - 상황공유 작성
  - 소식 작성
- 저장 항목
  - 제목
  - 설명
  - 장소 라벨
  - 상태 라벨
  - 생성 시각
  - 대표 이미지
  - 즐겨찾기 여부

## 공공데이터 확장 방향
- 저장된 장소가 TourAPI 장소와 매칭되면 공식 장소 정보로 보강한다.
- 향후에는 무장애, 반려동물, 장소 상세 메타를 재방문 정보에 연결할 수 있다.
- 나의 내발문자는 개인 기록 축이지만, 공공데이터가 재방문 가치 보강에 쓰일 수 있다.

## 확장 방향
- 장소 상세와 공공데이터 재매칭
- 사진 업로드
- 재방문 추천
- 지역별 추억 묶음

## 고도화 방향

### 1. 기록을 장소 단위로 묶기
- 현재는 상황공유와 소식이 각각의 기록으로 쌓이는 구조에 가깝다.
- 고도화 단계에서는 같은 장소, 같은 행사, 같은 TourAPI 콘텐츠를 하나의 `장소 기억 묶음`으로 보여준다.
- 예를 들어 수원화성에서 남긴 현장 공유, 후기, 즐겨찾기, 사진, 재방문 기록을 한 카드 안에서 이어서 볼 수 있어야 한다.
- 사용자는 개별 글 목록이 아니라 `내가 이 장소를 어떻게 경험했는지`를 회고하게 된다.

### 2. 지도 재진입을 기본 행동으로 만들기
- 나의 내발문자에서 가장 중요한 후속 행동은 `다시 보기`가 아니라 `지금 다시 확인하기`다.
- 기억 카드의 기본 CTA는 `지도에서 현재 상태 보기`로 둔다.
- 장소에 좌표가 있으면 지도 중심을 해당 장소로 이동하고, 최근 `live_status`와 주변 순간들을 함께 보여준다.
- 좌표가 없는 기록은 지도 재진입이 약하므로, 저장 시 가능한 한 장소명, 주소, 위도/경도를 함께 저장한다.

### 3. 발자취 타임라인 고도화
- 발자취는 단순 작성 시간순 목록이 아니라 사용자의 지역 경험 흐름을 보여주는 타임라인이어야 한다.
- 날짜별, 지역별, 장소 유형별로 묶어서 `언제 어디를 많이 다녔는지`가 보이게 한다.
- 기본 단위:
  - 오늘
  - 이번 주
  - 이번 달
  - 지역별
  - 행사별
- 타임라인 각 항목은 `지도에서 다시 보기`, `새 기록 남기기`, `즐겨찾기`로 이어진다.

### 4. 공식 관광데이터로 기억 보강
- 사용자가 저장한 장소가 TourAPI 콘텐츠와 매칭되면 공식 정보를 조용히 보강한다.
- 보강 항목:
  - 공식 장소명
  - 대표 이미지
  - 주소
  - 운영/행사 기간
  - 카테고리
  - 상세 설명 요약
- 단, 공식 정보가 카드의 주인공이 되면 안 된다. 주인공은 사용자의 기록이고, 공식 데이터는 재방문 판단을 돕는 보조 정보다.

### 5. 개인화 추천
- 나의 내발문자는 축적된 기록을 바탕으로 재방문 후보를 제안할 수 있다.
- 추천 기준:
  - 즐겨찾기한 장소
  - 여러 번 기록한 장소
  - 긍정적인 상태/후기가 많은 장소
  - 최근 행사가 열린 장소
  - 현재 `live_status`가 한산하거나 방문하기 좋은 장소
- 추천 문구는 과장하지 않고 행동 중심으로 둔다.
  - `전에 저장한 곳, 지금은 한산해요`
  - `최근 다시 열린 행사예요`
  - `지난번에 좋았던 장소 근처에 새 소식이 있어요`

### 6. 프로필/신뢰도 연동
- `public.profiles`가 생기면 나의 내발문자 상단에 개인 활동 요약을 붙인다.
- 표시 항목:
  - 닉네임
  - 함께한 시간
  - 신뢰도
  - 현장 공유 수
  - 확인받은 수
  - 소식 작성 수
  - 배지 레벨
- 이 정보는 과시용 랭킹보다 `내 기록이 지역에 얼마나 도움이 되었는지`를 보여주는 방향으로 사용한다.
- `함께한 시간`은 내발문자와 사용자가 함께한 기간 또는 누적 활동 시간을 의미한다.
  - 기본값은 `profiles.created_at`부터 현재까지의 기간으로 계산한다.
  - 향후 실제 체류/사용 로그가 생기면 앱 이용 시간 또는 활동 시간 누적으로 고도화한다.
  - 표시 예시는 `함께한 지 12일`, `이번 달 3시간 함께함`, `수원에서 5번 함께함`처럼 감정적이되 과장하지 않는다.

### 7. 기억 저장 방식 정리
- 별도 `album_memories` 테이블이 없으므로 초기에는 기존 데이터에서 나의 내발문자 화면을 구성한다.
- 상황공유는 `live_status`, 소식은 `posts`, 장소 정보는 `places`와 TourAPI 매칭값을 사용한다.
- 고도화 단계에서 독립 메모리 테이블이 필요해지면 `memories` 또는 `user_place_memories` 같은 이름으로 분리한다.
- 분리 시 필수 필드:
  - `user_id`
  - `source_type`
  - `source_id`
  - `place_name`
  - `address`
  - `latitude`
  - `longitude`
  - `tourapi_content_id`
  - `memory_type`
  - `is_favorite`
  - `created_at`

## 우선순위
1. 기록 카드에 좌표/주소/TourAPI ID를 최대한 보존한다.
2. 나의 내발문자 카드에 `지도에서 현재 상태 보기` CTA를 붙인다.
3. 상황공유와 소식을 장소 기준으로 묶는다.
4. 즐겨찾기와 발자취를 분리해서 보여준다.
5. TourAPI 매칭 정보로 대표 이미지와 공식 장소 정보를 보강한다.
6. `profiles`가 생기면 신뢰도/기여 요약을 상단에 붙인다.
7. 데이터가 충분히 쌓인 뒤 재방문 추천을 추가한다.

## UX 원칙
- 단순 보관함보다 `기억과 회고`의 느낌을 준다.
- 커뮤니티 기록과 개인 기록이 연결되어야 한다.
- 내가 남긴 기록이 다음 행동으로 이어지도록 `지도에서 다시 보기` 같은 CTA를 둔다.

## 성공 기준
- 사용자가 “내 기록이 쌓인다”는 감각을 느낀다.
- 즐겨찾기와 발자취가 실제 재방문 동기로 이어진다.
- 내발문자가 단발성 정보 앱이 아니라 지속형 서비스로 보인다.

## 흐름 연결 원칙
- 나의 내발문자는 `기억` 화면이다.
- 사용자가 남긴 상황공유와 소식은 가능한 한 장소 단위로 묶어 보여준다.
- 각 기억 카드에는 다음 행동이 필요하다.
  - `지도에서 다시 보기`
  - `새 기록 남기기`
  - `소식으로 보기`
  - `즐겨찾기`
- 기억한 장소에 좌표가 있으면 지도 재진입을 기본 후속 행동으로 둔다.
- 공식 TourAPI 장소와 매칭되면 재방문에 필요한 정보만 보강한다.
