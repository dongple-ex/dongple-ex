# 동플 허브 기능 개발 지침서

**목표:** 동플은 현장의 `지금`을 공유해 사람들의 선택을 바꾸는 실시간 상황 네트워크다. 한 사람의 순간이 모두의 선택이 되도록, 지도 기반 현재 상태 확인과 3초 상황 공유를 MVP의 중심에 둔다.

**한 줄 정의:** 한 사람의 순간이, 모두의 선택이 된다.

**최종 포지션:** 동플은 `타이밍을 알려주는 앱`, `확인하는 앱`, `지금 어디가 살아있는지 보여주는 앱`, 그리고 사람의 흐름을 바꾸는 서비스다.

---

## 1. 요약
- 가입: 초기 MVP는 간단한 사용자 식별과 기록 축적을 우선하고, 이후 휴대폰 본인인증(1인1계정) + 닉네임 설정으로 확장한다.
- 활동: 핵심 활동은 장소의 현재 상태를 빠르게 공유하는 것이다. 게시·제보·댓글의 익명성은 확장 단계에서 유지한다.
- 핵심 모듈: 지도 기반 상태 보기, 3초 상황 공유, 장소 상세, 마이페이지, 영향력 점수, Stream Service, Analytics
- 주요 목표: 실시간성 확보, 공유 속도 단축, 정확한 제보 유도, 사용자 행동 변화
- 제품 모토: **한 사람의 순간이, 모두의 선택이 된다.**
- 사용자 노출 원칙:
  - 전면 경험은 `지도를 본다 → 지금 상태를 확인한다 → 바로 공유한다 → 선택을 바꾼다` 흐름으로 구성한다.
  - 앱을 켜는 순간 사용자가 “오, 지금 여기 상황 바로 보이네?”라고 느껴야 한다.
  - 첫 화면은 `지금 핫한 곳`과 `내 주변 실시간 상태`이며, 지도 + 상태가 최우선이다.
  - 공유 UX는 최대 3초를 기준으로 하고, 버튼 중심·텍스트 최소화를 원칙으로 한다.
  - 공식 이벤트/지역 소식은 일정 정보가 아니라 `행사 현장의 현재 상태`를 만들기 위한 진입점으로 사용한다.
  - 홈 피드는 사용자 참여형 상황 카드와 공식 소식 카드를 혼합하되, 지도/상태 확인을 방해하지 않는다.

### 1-1. 핵심 가치
- **실시간성**: 과거 리뷰와 평균 데이터가 아니라 지금 이 순간의 현장 상태를 보여준다.
- **간결함**: 3초 안에 보고, 3초 안에 공유한다.
- **행동 변화**: 정보를 제공하는 데서 끝나지 않고 사용자의 이동, 방문, 대기, 예약 결정을 바꾼다.

### 1-2. MVP 범위
1. 지도 기반 상태 보기
   - 주변 장소 혼잡도 표시
   - 빨강/노랑/초록 상태 색상
   - 행사 레이어와 일반 장소 레이어 분리
2. 상황 공유
   - 혼잡도 선택
   - 태그 선택
   - 간단 코멘트
   - 즉시 반영
3. 장소 상세
   - 현재 상태
   - 최근 업데이트
   - 태그
   - 간단 코멘트
4. 마이페이지
   - 내 공유 기록
   - 영향력 점수
   - 배지

### 1-2-1. 행사/일상 레이어 전략
- 동플은 행사 일정 앱이 아니다.
- 행사 정보 자체보다 `지금 행사 어떤 상태냐`를 보여주는 서비스다.
- 행사는 초기 트래픽을 만드는 강한 진입점이다.
- 일상 장소는 행사가 없는 날에도 앱을 계속 쓰게 만드는 기본 레이어다.
- 제품 구조는 `행사 = 진입점`, `일상 = 확장`으로 둔다.
- 지도와 홈은 `오늘 행사`, `지금 가장 붐비는 곳`, `근처 한산한 곳`, `카페/식당/공원`을 함께 보여준다.
- 필터는 최소 `전체 / 행사 / 카페 / 식당 / 공원`을 제공한다.
- 핵심 문장은 **지금 가장 핫한 곳을 보여준다**로 확장한다.
- 운영 비유는 **행사 = 불 붙이는 장작, 일상 = 계속 타는 불**이다.
- 행사 데이터와 동플 데이터를 결합할 때의 핵심 구조는 **공공데이터 = 뼈대, 동플 = 실시간 뇌**다.
- 공공데이터는 이름, 위치, 시간, 설명을 제공하고, 동플은 지금 붐빔, 대기 상황, 분위기, 최근 업데이트를 제공한다.
- 지도 행사 핀은 강조하고, 색상으로 혼잡도를 보여주며, `몇 분 전` 업데이트인지 함께 노출한다.
- 예시 표현은 `벚꽃축제 - 지금 붐빔(5분 전), 줄 길어요, 사람 많음`처럼 현재 판단에 필요한 형태여야 한다.
- 후속 기능으로 `행사 상태 요약`을 제공할 수 있다.
  - 지금 매우 붐빔
  - 추천 시간: 저녁 8시 이후
  - 지금 가장 붐비는 축제

### 1-3. 제외 또는 후순위
- 찜 목록 중심 첫 화면
- 커뮤니티 글 중심 첫 화면
- 긴 설명 중심 랜딩
- 공유 직후 과도한 보상 연출
- 초기부터 전국 확장

## 2. 시스템 아키텍처 개요
- Frontend: Next.js
- Backend: Supabase
- 상태관리: TanStack Query 도입을 목표로 한다.
- 지도: 네이버/카카오 지도를 우선 검토한다.
- 앱화: 웹으로 먼저 만들고 Capacitor 기반 앱 확장을 후속 단계로 둔다.
- Stream Service: MVP에서는 Supabase Realtime 또는 주기적 재조회로 시작하고, 이후 WebSocket/SSE와 Redis Pub/Sub 캐시로 확장한다.
- Analytics Engine: 최근 30분 기준 상태 계산, 태그 빈도, 영향력 점수를 우선 구현하고, 이후 집계·트렌드 계산으로 확장한다.
- Auth & Anonymity Layer: MVP는 사용자 기록과 정확도 추적을 우선하고, 이후 본인인증 서비스 연동, 익명화 해시, 운영 식별자 매핑으로 확장한다.
- Admin Console: 제재·검증·로그·통계 확인은 데이터 밀도 형성 후 확장한다.

### 2-1. 확장 아키텍처
- Collector: 뉴스 RSS, SNS(X) 검색, 동플 게시물, 사용자 제보, 공공데이터 수집
- Message Queue: Kafka / RabbitMQ (이벤트 버퍼링, 확장성)
- Classifier: 텍스트 전처리 → 카테고리 분류 → 위치 태깅 → 중복 병합 → 신뢰도 산정
- DB 확장: PostgreSQL(주 데이터), ElasticSearch(검색/유사도), Redis(캐시)

## 3. 데이터 모델(요약 ERD)

> 현재 저장소 기준 메모
> - 이 문서는 목표 아키텍처와 확장 모델을 포함한 마스터 스펙이다.
> - 현재 폴더의 `schema.sql`은 이 스펙의 전체 구현이 아니라 일부 MVP/초기 구현 범위를 반영한다.
> - 현재 SQL 기준으로 확인 가능한 테이블은 `live_status`, `status_verifications`, `posts`, `events`, `events_ext`, `raw_items`이다.
> - v1 기획 기준의 핵심 모델은 `users`, `places`, `reports`, `place_status`다.
> - `profiles`, `event_links`, `stats_daily`는 확장 모델이며, 현재 SQL에는 아직 반영되지 않았다.

### 핵심 테이블 요약

| 테이블 | 주요 필드 | 설명 |
|---|---:|---|
| **users** | id; nickname; created_at; trust_score | 사용자와 공유 기록의 기준 |
| **places** | id; name; category; address; lat; lng; region | 상태를 확인하고 공유하는 장소 |
| **reports** | id; user_id; place_id; crowd_level; tags; comment; created_at | 사용자의 3초 상황 공유 원본 |
| **place_status** | place_id; current_level; report_count; top_tags; updated_at; expires_at | 최근 30분 기준으로 계산된 현재 상태 |
| **events** | id; title; content; category; source; location_name; lat; lng; created_at; score | 공식 행사/공공데이터 스트림 항목 |
| **raw_items** | id; source; raw_text; fetched_at; processed_flag | 수집 원문(수집 로그) |

### 행사 상태 결합 모델
행사 상태는 공공 API의 행사 메타와 동플 실시간 공유 데이터를 결합해 만든다.

| 축 | 데이터 | 역할 |
|---|---|---|
| 공공 행사 데이터 | event_id, 이름, 위치, 시간, 설명, 이미지 | 행사 상태 카드의 기본 뼈대 |
| 동플 리포트 | event_id, crowd_level, wait_tag, mood_tag, comment, created_at | 행사 현장의 실시간 판단 정보 |
| 행사 상태 요약 | event_id, current_level, top_tags, last_reported_at, report_count | 지도/홈에 노출할 현재 상태 |

기술 흐름은 다음과 같다.

1. 행사 API에서 일정과 위치를 가져온다.
2. `events` 테이블에 저장하고 위치를 매핑한다.
3. 지도에 강조 행사 핀을 표시한다.
4. 사용자의 상황 공유를 `event_id` 기준으로 연결한다.
5. 최근 공유를 기반으로 행사 현재 상태와 요약을 계산한다.

### 3-1. 현재 상태 계산 로직
- 최근 30분 내 `reports`를 기준으로 장소별 평균 혼잡도를 계산한다.
- 혼잡도는 `붐빔`, `보통`, `한산` 세 단계로 우선 단순화한다.
- 태그는 빈도 기반으로 상위 항목을 `place_status.top_tags`에 반영한다.
- 최근 데이터가 없으면 상태를 만료하거나 신뢰도를 낮춰 표시한다.

## 4. 인증·익명화 모듈 설계

### A. 가입 및 인증 흐름
1. 휴대폰 본인인증(외부 인증사 연동) → phone_hash 저장(해시화)
2. 계정 생성 → 닉네임 설정(중복 허용 가능)
3. 동네 인증: GPS 기반 또는 우편/행정동 선택(선택적)
4. 가입 완료 후 활동 가능

### B. 익명화 원칙(Anonymity Layer)
- 운영 식별자(user_id)는 내부 DB에만 존재. 외부/프론트엔드에는 절대 노출 금지.
- 게시물/제보 저장 시 public_id 생성: `public_id = HMAC(secret_key, user_id + salt + timestamp_bucket)`
  - public_id는 동일 사용자의 연속 활동을 연결할 수 있으나 실제 user_id 복원 불가(운영자만 매핑 가능).
- 완전 익명 옵션: 사용자가 “완전 익명” 선택 시 public_id 대신 `anonymous` 태그만 노출. 운영자는 내부 로그로 추적 가능.
- 로그 보관 정책: 본인인증 관련 민감 데이터는 암호화 저장, 접근 권한 최소화, 법적 요청 시만 복호화 가능.

### C. 위치 인증
- 가입 시 GPS 허용 권장. GPS 미허용 시 사용자가 직접 행정동 선택.
- 제보 시 위치 자동 추출(텍스트) + 사용자가 위치 수정 가능.
- 위치 조작 탐지: 동일 계정의 빈번한 지역 변경 시 경고/검증 요청.

## 5. 자동 분류·신뢰도 알고리즘 (Classifier & Scoring)

v1 기준의 우선 알고리즘은 `정확한 공유 → 현재 상태 계산 → 영향력 피드백`이다. 자동 분류와 외부 데이터 기반 신뢰도 산정은 확장 단계로 둔다.

### MVP 상태 계산
1. 사용자가 장소를 선택한다.
2. 혼잡도(붐빔/보통/한산)를 선택한다.
3. 태그와 선택적 코멘트를 남긴다.
4. 최근 30분 내 같은 장소의 공유를 모아 현재 상태를 계산한다.
5. 지도와 장소 상세에 즉시 반영한다.

### 영향력 점수
- 보상은 많이 올린 사람이 아니라 정확하게 올린 사람에게 집중한다.
- 영향력 점수의 의미는 `내 정보로 몇 명이 움직였는가`다.
- 공유 직후 즉시 보상보다, 나중에 조회·방문·상태 확인 흐름을 기반으로 영향 피드백을 제공한다.
- 초기에는 조회 수, 상태 확인 수, 후속 공유 일치율을 점수 후보로 둔다.

### 초기 성장 데이터 전략
- 한 지역에 집중한다.
- 직접 데이터를 채워 `앱을 켜면 정보가 있음` 상태를 만든다.
- 친구 참여와 하루 20~30개 등록을 초기 실행 기준으로 둔다.
- 초기 콘텐츠는 행사와 팝업처럼 사람이 몰리는 지점을 우선 활용한다.
- 예: `오늘 홍대 상황`, `지금 성수 팝업 몰림`, `수원 행사장 현재 대기`.
- 단, 콘텐츠의 주제는 행사 일정이 아니라 행사 현장의 현재 상태여야 한다.

### A. 처리 파이프라인
1. 수집(raw_items) → 텍스트 정제(정규화, 불용어 제거)
2. 언어 처리: 형태소 분석(한국어), 개체명 인식(NER)으로 장소·시간·인물 추출
3. 카테고리 분류: 하이브리드 모델
   - 룰 기반 키워드 매칭(긴급 카테고리 우선)
   - ML 분류기(KoBERT/KoELECTRA)로 문맥 분류
4. 위치 매핑: 추출된 장소명 → 행정동/좌표 매핑(지오코딩 DB)
5. 중복 병합: 시간 윈도우(예: 30분), 위치 반경(예: 500m), 텍스트 유사도(ElasticSearch) 기준 병합
6. 이벤트 생성: events 테이블에 저장

### B. 신뢰도(Score) 산정
- 기본 가중치: News 0.9; Official Public Data 0.95; SNS 0.7; 동플 게시물 0.6; 사용자 제보 0.5
- 가중치 조정 요소:
  - 출처 수(동일 사건을 보고한 출처 수) → +0.05 per unique source (cap 0.2)
  - 시간 신선도(최근성) → 최근일수록 가중치 증가
  - 사용자 평판(제보자 과거 신뢰도) → +/− 보정
  - 중복 병합 횟수(여러 사용자가 독립 제보) → 신뢰도 증가
- 최종 score는 0.0~1.0 범위로 정규화하여 스트림 우선순위 및 통계 반영에 사용

### C. 허위·악성 탐지 룰
- 동일 텍스트 반복 제출(스팸) → 자동 플래그
- 위치와 텍스트 불일치(예: 서울 관련 텍스트에 경기 좌표) → 검증 요청
- 다수 신고 발생 시 자동 임시 숨김 후 관리자 검토
