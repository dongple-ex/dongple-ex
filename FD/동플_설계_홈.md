# 동플 설계 - 홈

## 역할
- 동플의 첫 진입 화면
- 실시간 상황과 공식 소식을 가장 빠르게 이해시키는 허브
- 지도와 기록으로 사용자를 보내는 분기점

## 핵심 목표
- `지금 우리 동네 상황`을 먼저 보여준다.
- `공식 소식`은 보조 축으로 함께 보여준다.
- 사용자가 바로 `지도 보기` 또는 `기록하기`로 이동할 수 있어야 한다.

## 주요 구성
- 상단 요약 헤더
  - 동플의 현재 가치 제안
  - 지도 진입 CTA
  - 기록 진입 CTA
- 실시간 상태 요약
  - 최근 상황공유 카드
  - 상태 분포 또는 대표 상태
- 지도 허브 안내
  - 지도에서 전체 상황 보기
- 공식 소식 요약
  - 행사, 축제, 공식 안내
- 보조 커뮤니티 흐름
  - 필요 시 소식 미리보기

## 노출 비율 원칙
- 초기 화면 비중은 `상황 60~75%`, `공식 소식 25~40%`를 기준으로 둔다.
- 소식 커뮤니티 피드는 홈의 하단 보조 섹션으로 둔다.
- 홈은 홍보성 랜딩이 아니라 현재 동네 상태를 읽는 시작점처럼 보여야 한다.

## 제외 원칙
- 홈을 일반적인 홍보 랜딩처럼 만들지 않는다.
- 긴 설명, 과도한 프로모션 배너, 무관한 카드 나열은 줄인다.
- 소식 피드는 홈의 주인공이 아니라 보조다.

## 데이터 축
- 사용자 데이터
  - 상황공유
  - 최근 기록
- 공공데이터
  - 공식 행사
  - 관광공사 기반 관광 정보 요약

## 구현 관점
- `app/page.tsx`는 화면 조립을 담당한다.
- 실시간 상태, 공식 소식, 커뮤니티 미리보기는 각 feature 단위 컴포넌트로 유지한다.
- 홈에서 보여주는 공식 소식은 TourAPI 기반 공식 데이터 우선, 실패 시 fallback 데이터를 사용한다.

## 성공 기준
- 첫 화면만 보고도 `상황 중심 서비스`라는 인상이 전달된다.
- 지도 진입 의도가 바로 생긴다.
- 공식 소식이 실시간 축을 방해하지 않는다.

## 관광자료 활용 방향
- 동플은 전국 유명 관광지 랭킹을 보여주는 서비스보다, 사용자의 현재 위치와 가까운 지역 관광 자원을 더 잘 활용하게 만드는 서비스에 가깝다.
- 따라서 홈에서 보여주는 공식 관광 자료도 `멀리 있는 대표 명소`보다 `지금 내 주변에서 바로 참고할 수 있는 행사, 장소, 생활권 정보`를 우선한다.
- 이 원칙은 동플의 지역 밀착형 정체성을 강화하고, 관광과 생활이 겹치는 실제 사용 맥락을 더 잘 반영한다.
