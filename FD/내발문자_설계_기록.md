# 내발문자 설계 - 기록

## 역할
- 전역 작성 허브
- `상황공유`와 `소식 작성`을 하나의 입구에서 선택하게 하는 공통 액션

## 핵심 목표
- 사용자가 페이지마다 다른 작성 버튼을 학습하지 않도록 한다.
- 현재 맥락에 맞는 기본 탭을 제안하되, 항상 다른 탭으로 전환할 수 있게 한다.

## 기본 구조
- 진입 명칭
  - `기록하기`
- 탭
  - `상황공유`
  - `소식 작성`
- 기본 탭 규칙
  - `/map`에서 열면 `상황공유`
  - `/news`에서 열면 `소식 작성`

## 작성자 컨텍스트
- 작성자는 익명 또는 닉네임 기반으로 활동한다.
- 기록 화면에서는 현재 작성 정체성을 명확하게 보여준다.
- 향후 프로필/인증 체계가 확장되어도 `기록` 자체의 진입 구조는 유지한다.

## 상황공유 성격
- 즉시성 중심
- 짧은 입력
- 위치 맥락 필수
- 현장 상태, 분위기, 대기, 요청

## 소식 작성 성격
- 맥락 중심
- 게시글 구조
- 제목, 본문, 카테고리
- 질문, 추천, 생활 정보, 후기

## 데이터 연결
- 상황공유 작성 후
  - 실시간 상태 보드 반영
  - 지도 마커 반영
  - 나의 내발문자에 메모리 저장
- 소식 작성 후
  - 소식 목록 반영
  - 나의 내발문자에 메모리 저장

## 운영 확장 포인트
- 신고, 검증, 신뢰도 보정은 기록 이후 후속 액션으로 연결된다.
- 기록 원본은 사용자 축 데이터이며, 공식 데이터와 직접 합쳐 저장하지 않는다.

## UI 원칙
- 상단 공간은 과도하게 크지 않게 유지한다.
- 입력 폼이 첫 화면에서 충분히 보이도록 한다.
- 작성 완료 후 흐름이 자연스럽게 닫히거나 후속 이동으로 이어져야 한다.

## 성공 기준
- 기록 버튼의 의미가 흔들리지 않는다.
- 두 작성 방식의 차이가 탭 구조만으로도 이해된다.
- 작성한 내용이 이후 `나의 내발문자`에 누적되어 연결된다.

## 완료 후 흐름
- 기록은 작성 자체로 끝나지 않는다.
- 상황공유 완료 후:
  - `지도에 반영됨` 피드백을 보여준다.
  - 후속 CTA는 `지도에서 보기`, `나의 내발문자에서 보기`를 우선한다.
- 소식 작성 완료 후:
  - `소식에 올라감` 피드백을 보여준다.
  - 후속 CTA는 `소식에서 보기`, `나의 내발문자에서 보기`를 우선한다.
- 기록 저장 시 위치 정보가 있으면 반드시 나의 내발문자 메모리에도 함께 저장한다.
  - 장소명
  - 주소
  - 위도/경도
  - 공식 장소 매칭 ID
  - 상태 라벨 또는 게시글 카테고리
