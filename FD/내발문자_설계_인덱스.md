# 내발문자 설계 인덱스

## 목적
- 현재 내발문자의 기능 구조를 실제 서비스 메뉴 기준으로 다시 정리한다.
- 앞으로의 설계, 구현, QA, 공모전 설명이 같은 분류를 보도록 맞춘다.
- 기존 FD 문서는 참고 아카이브로 두고, 이 문서 세트를 운영 기준으로 삼는다.
- `내발문자_기획서_v1.md`의 방향을 우선 반영한다.
- 내발문자는 `dongple_ex`에서 동플의 확장 버전으로 출발했지만, 이후 문서와 구현에서는 별도 서비스명으로 다룬다.

## 제품 정의
- 한 줄 정의: `내 발이 남긴 문장, 다음 발걸음의 선택이 된다.`
- 내발문자는 현장의 지금과 사용자의 발자취를 연결해 사람들의 다음 선택을 바꾸는 실시간 관광 상황 네트워크다.
- 핵심 포지션은 `타이밍을 알려주는 앱`, `확인하는 앱`, `지금 어디가 살아있는지 보여주는 앱`이다.
- 최우선 경험은 `지금 핫한 곳`과 `내 주변 실시간 상태`를 지도에서 바로 확인하는 것이다.
- 성장 구조는 `행사 = 진입점`, `일상 = 확장`이다.
- 행사 결합 구조는 `공공데이터 = 뼈대`, `내발문자 = 실시간 뇌`다.

## 공통 원칙
- 홈은 `지금 핫한 곳`과 `내 주변 실시간 상태`를 보여주는 첫 진입점이다.
- 지도는 `행사 레이어`와 `일반 장소 레이어`를 함께 다루는 실시간 탐색과 공유의 핵심 화면이며, 찜 목록이나 커뮤니티 글보다 우선한다.
- 기록은 `상황공유`와 `소식 작성`을 아우르는 공통 진입점이다.
- 소식은 `커뮤니티형 게시판`이다.
- 나의 내발문자는 `기록의 축적, 회고, 재방문`을 다루는 개인 영역이다.
- 공유 UX는 3초 안에 끝나야 한다.
- 상태 표현은 우선 `붐빔 / 보통 / 한산` 세 단계로 단순화한다.
- 내발문자는 정보를 보여주는 데서 끝나지 않고 사용자의 선택을 바꾸는 것을 목표로 한다.
- 행사 정보 자체가 아니라 행사의 현재 상태를 보여준다.

## 구현 아키텍처 원칙
- `app/*`는 조립과 라우팅의 책임을 가진다.
- 실제 도메인 로직은 `features/*`와 `services/*`로 분리한다.
- UI 컴포넌트와 데이터 접근 로직은 분리한다.
- 공식 데이터와 사용자 데이터는 같은 화면에 보여도 저장 축은 분리한다.
- 초기 핵심 데이터 축은 사용자 축, `places`, `live_status`, `status_verifications`다.
- 사용자 축은 현재 Supabase Auth 사용자 또는 익명/닉네임 작성자 정보로 처리하며, 영향력 점수, 배지, 신뢰도, 프로필이 필요해지면 `public.profiles` 테이블을 추가한다.
- 사용자 축을 테이블로 확장할 때는 `public.users`가 아니라 `public.profiles`를 사용하며, 생성 SQL은 [scripts/create-profiles.sql](/C:/my_project/dongple_ex/scripts/create-profiles.sql)에 둔다.
- 상태 계산은 최근 30분 데이터와 태그 빈도 기반으로 시작한다.
- `live_status`는 사용자 상황공유 원본 테이블이다.
- `place_status`는 현재 필수 테이블이 아니며, 장소별 현재 상태 요약이 필요할 때 집계 결과 또는 향후 캐시 테이블로 다룬다.
- 검증 데이터는 `status_verifications`로 분리한다.

## 전역 내비게이션 원칙
- 하단 메뉴의 기준 축은 `홈 / 지도 / 기록 / 소식 / 나의 내발문자`이다.
- `기록`은 단일 기능이 아니라 `상황공유`와 `소식 작성`으로 이어지는 허브다.
- `나의 내발문자`는 개인 기능의 상위 개념이며, 앨범은 그 안의 주요 섹션이다.
- 첫 화면은 커뮤니티 글이나 찜 목록이 아니라 지도와 상태가 중심이어야 한다.
- 전체 사용자 흐름은 `홈에서 발견 → 소식으로 맥락 이해 → 기록으로 남김 → 지도에서 위치/상태 확인 → 나의 내발문자에서 기억 관리`로 이어져야 한다.

## 브랜드 확장 원칙
- `내발문자`는 코어 서비스 브랜드다.
- 지역 론칭 시에는 `내발문자: 수원에 가다`, `내발문자: 강원삼척에 가다`처럼 지역 테마를 얹을 수 있다.
- 지역 테마는 브랜딩과 큐레이션 레이어이며, 제품 구조 자체는 공통 코어를 유지한다.

## 공공데이터 원칙
- 한국관광공사 OpenAPI는 공식 관광 데이터 축이다.
- 사용자 생성 데이터와 공공데이터는 같은 화면에 공존할 수 있지만 성격은 분리해서 설명한다.
- 공공 행사 데이터는 일정 앱을 만들기 위한 것이 아니라 현장 상태 공유를 붙이는 진입점이다.
- 행사 화면과 지도는 행사 이름/위치/시간보다 `지금 붐빔`, `대기 상황`, `분위기`, `몇 분 전`을 우선 노출한다.
- 운영 기준은 [내발문자_TourAPI_연동운영_가이드.md](/C:/my_project/dongple_ex/FD/내발문자_TourAPI_연동운영_가이드.md)를 따른다.

## 문서 운영 규칙
- 새로운 기능은 먼저 어느 메뉴 축에 속하는지 결정한 뒤 해당 설계 문서에 반영한다.
- 화면 수정이 메뉴 경계를 넘으면 인덱스와 관련 설계 문서를 함께 갱신한다.
- 이전 구상 문서는 삭제하지 않고 참고용으로 유지한다.

## 기존 문서에서 흡수한 핵심 항목
- 프론트 구조 문서의 `app / features / services` 분리 원칙
- 시스템 설계 문서의 `상황공유 / 커뮤니티 / 공식 이벤트 / 인증 / 운영` 모듈 구분
- 지도/상태보드 보완 문서의 `바텀시트`, `드래그`, `지도-리스트 연동`, `즉시 액션` UX
- TourAPI 문서의 `Ground Truth`, `fallback`, `메타 보강`, `나의 내발문자` 확장 관점
- 기획서 v1의 `3초 공유`, `지도 + 상태 첫 화면`, `영향력 점수`, `한 지역 집중` 전략
- 행사/일상 전략의 `행사 = 진입점`, `일상 = 확장`, `지금 가장 핫한 곳` 포지션
